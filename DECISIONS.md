# Decisions

## 2026-04-04: /start всегда пересоздаёт сессию — баг

**Проблема:** `/start` в группе всегда создаёт пустую `GroupSession`, стирая всех добавленных пилотов. Если `/start` вызван после `/add`, пилоты теряются.

**Диагноз из скриншота:** Все команды `/start`, `/add`, "▶️ Старт" отправлены в 20:30. `/start` обработался после `/add` и обнулил `Tracking`.

**Решение:** `/start` в группе должен проверять, существует ли сессия:
- Если сессии нет → создать новую
- Если сессия есть и пилоты есть → спросить "Продолжить / Новая сессия" (reuse `askStartChoice`)
- Если сессия есть, но пуста → ответить "Сессия уже активна"

Так же нужен `/start_session` для явного создания свежей сессии.

## 2026-04-04: OGN клиент не подключается после execTrackOn

**Проблема:** `execTrackOn` ставил `TrackingOn = true` → вызывал `updateFilter()` → тот видел `TrackingOn == true` и вызывал `Disconnect()` → `killed = true` навсегда → `runClient` запускался, но `Run()` видел `killed` и возвращал nil → бесконечный цикл без подключения.

**Решение (v1, неполное):**
1. В `execTrackOn`: вызывать `updateFilter()` ДО `s.TrackingOn = true`, чтобы filter обновился без Disconnect
2. В `updateFilter`: при `TrackingOn == true` — полный перезапуск горутин (close StopCh + Disconnect + новые горутины)

**Проблема v1:** `Disconnect()` ставит `killed = true` навсегда в `*client.Client`. Новый `Run()` на том же клиенте видит `killed == true` и выходит мгновенно. Та же проблема возникает при повторном `execTrackOn` после `stopTracking` (который тоже вызывает `Disconnect`).

**Решение (v2):** После каждого `Disconnect()` создавать новый `client.New(...)` вместо переиспользования убитого клиента. Применено в:
- `updateFilter()` — mid-tracking рестарт при изменении фильтра
- `execTrackOn()` — старт трекинга после предыдущего стопа

## 2026-05-07: Закрепление summary через PinChatMessage

**Проблема:** summary-сообщение (список пилотов, обновляемое каждые 30 с) уплывает вверх чата при любой активности группы или при отправке свежей live-локации нового пилота. Пользователю приходится скроллить, чтобы найти текущее состояние.

**Рассмотренные варианты:**
- **Delete + resend на каждом тике с внешней активностью.** Сохраняет «summary снизу», но меняет `MessageID` на каждом ресенде (inline-кнопки `pickup:<id>` пришлось бы пере-биндить), 2 API-вызова на тик, push-уведомление на каждом ресенде даже с `disable_notification` (Telegram считает send новым событием). Под чатным флудом — заметные tеле API-вызовы и шум.
- **`PinChatMessage` с `DisableNotification=true`.** Закрепляет сверху чата (баннер под заголовком), `MessageID` стабилен (callbacks работают без перебиндов), один API-вызов один раз за сессию, без push и без service-сообщения «Bot pinned a message».

**Выбрано:** pin. Trade-off — summary живёт сверху, а не снизу, но это компенсируется тем, что он всегда виден без скролла и не плодит мусор в истории чата.

**Реализация:**
- Закрепляем после первого пришедшего бикона (`withPos > 0`), чтобы не пинить пустую «жду данных» карточку.
- На завершении сессии (`/track_off`, `/session_reset`, `/debug_wipe`, повторный `/track_on`) — открепляем (`UnpinChatMessage` в детачнутой горутине, чтобы не блокировать caller под `t.mu`).
- `SummaryMsgID` и `SummaryPinned` персистятся, чтобы рестарт бота не приводил к повторному пину уже-закреплённого сообщения и не давал дубль push-уведомления.
- Если бот лишён `can_pin_messages`, pin фейлится → warn в логи → следующий тик ретраит автоматически. Когда право выдают — pin срабатывает и `SummaryPinned=true` фиксирует факт навсегда.

**Требование к деплою:** бот должен быть админом группы с разрешением `can_pin_messages`.

## 2026-05-07: Эфемерный cleanup транзитных сообщений

**Проблема:** каждая операция (/start, /add, /track_on, нажатие reply-кнопки) оставляет в чате пару «команда + ack» — за серию действий чат превращается в простыню транзитного шума, среди которого теряется реально полезный контент (live-локации пилотов, landing-алерты).

**Реализация:**
- 5-секундная задержка после финального ack, потом батч `deleteMessages` (один API-вызов на всю цепочку).
- Single-step команды (например, `/track_on`): exec возвращает `ackMsgID`, caller вызывает `scheduleEphemeralDelete(chatID, userMsgID, ackMsgID)`.
- Multi-step флоу (confirm-диалоги `/track_off`/`/session_reset`, `/add` без аргумента → DM bridge, `/landing` / `/driver` / `/area` ждущие локацию, кнопка «📡 Радиус» с цифровым вводом): промежуточные транзитные ID накапливаются в `GroupSession.PendingCleanup map[userID][]int`, на финальном шаге дренем + аппендим финальный ID + `scheduleEphemeralDelete` всем батчем.
- Игнорим в cleanup: summary, live-локации пилотов, landing-алерты, ответы info-команд (`/list`, `/help`, `/status`, `/myid`), DM-история, геолокация-пин самого пользователя в `/landing`/`/area`/`/driver` (она — полезный артефакт).

**Что НЕ делаем:**
- Очередь `PendingCleanup` runtime-only, не персистится. Рестарт бота посреди ожидающего флоу → транзитные сообщения остаются в чате; приемлемая деградация, не критично.
- Нет персистентной очереди отложенных удалений — таймер живёт только в горутине, переживёт лишь до рестарта.
- Нет undo. Удалили — значит удалили.

**Edge cases:**
- Бот без `can_delete_messages` → свои сообщения удалит, чужие нет (DeleteMessage вернёт ошибку → warn в лог). Команды юзеров останутся.
- Сообщение уже удалено вручную → `"message to delete not found"` → warn, идём дальше.
- Параллельные флоу разных пользователей не пересекаются — очередь keyed по `cb.From.ID` / `m.From.ID`.

**Требование к деплою:** дополнительно к `can_pin_messages` боту нужно `can_delete_messages`.
